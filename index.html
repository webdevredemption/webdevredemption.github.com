First Idea: project.json

What is it?

A single json file that is used by all front-end development tools to access a project's structure information as well as configuration. More info

Why should it be used?

Creating a standard method of accessing the properties and configuration of a project now will allow the current tools to be used with existing projects (if a project.json file is created for that project) and ensure that there is a standard for the future development of front-end tools.

How will it expand/extend with new tools?

Being json the file will be very easy to expand, a sample file is located in the repository and be used as a proof of concept. simply adding the require file checking to a new tool and creating a new object in an existing project.json is all that is required to expand.

Any proof of this in practice?

Due to the early stages of the idea I am yet to implement this theory in to practice, however I have plans to do so in the near future (see forked repositories).
